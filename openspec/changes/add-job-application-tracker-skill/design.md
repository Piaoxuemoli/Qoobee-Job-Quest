## Context

当前仓库以 README 作为投递记录主入口，但缺少稳定的机器维护边界与统一字段，导致新增/修改后统计容易不一致。该变更目标是在不引入复杂基础设施的前提下，定义可落地的最小方案：通过对话收集结构化字段，更新 README 指定区块，并可选执行 Git 同步。

约束条件：
- 仅维护单仓库内记录，不依赖外部数据库。
- 以 README 为唯一展示面，打开仓库即可查看。
- 进度为非线性流程，允许跳跃与提前结束。

## Goals / Non-Goals

**Goals:**
- 定义统一记录模型（字段、ID、枚举、校验规则）。
- 定义新增与修改两条对话 upsert 流程。
- 定义 README 机器维护区块与统计重算规则。
- 定义 Git 同步策略（显式确认、提交规范、失败处理）。

**Non-Goals:**
- 不设计多用户并发编辑与锁机制。
- 不引入数据库、Web UI 或服务端 API。
- 不做复杂历史分析报表（如漏斗、转化率）作为 MVP 必选项。

## Decisions

### Decision 1: README 采用双机器维护区块
- 选择：在 README 中定义 `SUMMARY` 与 `RECORDS` 两个标记区块，仅更新区块内内容。
- 原因：最大化兼容人工撰写内容，降低误改风险；也便于后续脚本稳定定位。
- 备选方案：整体重写 README。
- 未选原因：会覆盖手写段落，冲突与回滚成本高。

### Decision 2: 数据模型以“当前进度”作为主统计口径
- 选择：每条记录仅维护一个 `currentProgress`（当前进度），统计按当前值计算。
- 原因：实现最小、可解释性强，满足“打开即看当前状态”的核心诉求。
- 备选方案：仅维护完整进度历史并动态推导当前进度。
- 未选原因：实现复杂度更高，且对 MVP 价值有限。

### Decision 3: 记录定位优先级为 ID > 公司+岗位+时间
- 选择：修改流程优先使用唯一 ID；无 ID 时再使用组合键兜底。
- 原因：减少歧义，避免同公司多岗位或重复投递冲突。
- 备选方案：仅依赖公司名定位。
- 未选原因：高冲突概率，修改风险大。

### Decision 4: 进度枚举固定且允许非线性流转
- 选择：固定枚举为“已投递、测评、HR联系、笔试、一面、二面、三面、四面、面委会、HR面、Offer、拒绝、主动中止”，允许跳跃。
- 原因：统一口径并适配不同公司流程差异。
- 备选方案：强制线性状态机。
- 未选原因：不符合真实招聘流程，误拦截合法更新。

### Decision 5: Git 同步采用显式确认
- 选择：每次更新后询问是否执行 `add/commit/push`。
- 原因：避免误推送，保持用户可控。
- 备选方案：自动推送。
- 未选原因：风险高，且不利于用户审阅变更。

## Risks / Trade-offs

- [Risk] README 标记区被手动破坏导致无法定位更新区块 → Mitigation: 更新前检测标记完整性，缺失时给出修复提示模板。
- [Risk] 无 ID 场景下组合键可能匹配多条记录 → Mitigation: 回显候选列表并要求二次确认。
- [Risk] 进度枚举后续扩展导致历史记录混用旧值 → Mitigation: 维护单一枚举清单并在写入时强校验。
- [Risk] push 失败造成“本地已更新、远端未更新” → Mitigation: 明确报告失败并保留本地提交供后续重试。

## Migration Plan

1. 在 README 初始化双机器维护区块与明细表头。
2. 引入记录模型与校验规则，先支持新增流程。
3. 增加修改流程与记录定位逻辑（ID 优先）。
4. 接入统计重算逻辑并验证进度、类型计数。
5. 接入 Git 显式同步流程与失败回报。
6. 回滚策略：保留 README 与本地提交历史，必要时通过 `git revert` 撤销本次变更。
